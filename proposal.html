Konexa CV Parser Web

NestJS · React · PostgreSQL · Microsoft OAuth2 · Teamtailor · PM2 bare metal · Traefik

Visión General

Plataforma web interna para el equipo de reclutamiento de Konexa. Centraliza la extracción, edición y exportación de CVs en formato Konexa-branded, con acceso al ATS Teamtailor, sin depender de instalaciones locales ni de Microsoft Word. Se despliega en el servidor existente junto a los demás servicios Node.js gestionados con PM2.

📄 Parsing Flexible

Sube uno o varios CVs manualmente (PDF/DOCX) o impórtalos desde Teamtailor. Extracción con GPT-5-mini, editor interactivo y descarga del PDF Konexa-branded en el mismo flujo.

🔗 Buscador Teamtailor

Busca candidatos por nombre o por ID con detección automática del modo. Debounce de 350ms para no saturar la API. Selecciona e importa el CV al pipeline con un solo clic.

🔐 Acceso Corporativo

Login exclusivo con cuenta Microsoft de Konexa (OAuth2 / Azure Entra ID). Sin contraseñas independientes. API Key de OpenAI compartida, gestionada en el servidor.

Principio central: Todo el procesamiento ocurre en el servidor — conversión de DOCX, llamadas a OpenAI, descarga de CVs desde Teamtailor y generación de PDF. El frontend es consumidor puro de la API REST. El PDF se genera bajo demanda y se entrega directamente al browser, sin guardarlo en servidor.

Autenticación Microsoft OAuth2 (Azure Entra ID)

Acceso restringido a cuentas del tenant de Konexa en Microsoft 365. El flujo usa OAuth2 Authorization Code + PKCE. Se usará una App Registration nueva y dedicada a este sistema.

1. Browser
Clic en "Entrar con Microsoft"
Recibe el JWT propio de Konexa
JWT en memoria (React Context). Axios interceptor añade Bearer token a cada petición
2. Microsoft Entra ID
Muestra login corporativo del tenant Konexa
Valida la cuenta. Solo acepta emails del tenant configurado
Retorna id_token con nombre, email y oid (Object ID de Azure)
3. NestJS
GET /api/auth/microsoft → redirect a MS
GET /api/auth/microsoft/callback → valida token con passport-azure-ad. Crea usuario en BD si es nuevo (primer login)
Emite JWT propio firmado con JWT_SECRET. Válido en cada petición posterior

Integración Teamtailor — Buscador con Detección Automática de Modo

Teamtailor actúa únicamente como fuente de CVs. El sistema no escribe datos en TT. El buscador usa debounce de 350ms para no saturar la API, y detecta automáticamente si el usuario está buscando por nombre o por ID con un regex simple y determinista.

🔍 Lógica de detección: nombre vs. ID

La distinción se hace evaluando el input con el regex /^(TT-?)?\d+$/i antes de hacer el llamado a la API de Teamtailor:

Lo que escribe el usuario¿Coincide con regex?Modo detectadoParámetro API de TT
CarlosNo🔵 Nombrefilter[name]=Carlos
María GonzálezNo🔵 Nombrefilter[name]=María González
4821✅ Sí — solo dígitos🟡 IDfilter[id]=4821
TT-4821✅ Sí — prefijo TT-🟡 ID (extrae 4821)filter[id]=4821
TT4821✅ Sí — prefijo TT🟡 ID (extrae 4821)filter[id]=4821
123abcNo — letras mezcladas🔵 Nombrefilter[name]=123abc

Además, el frontend muestra un pill indicador de modo que cambia en tiempo real mientras el usuario escribe, antes incluso de que se dispare el request:

// frontend: useSearchMode hook
const ID_REGEX = /^(TT-?)?\d+$/i;

function detectMode(q: string): 'default'|'name'|'id' {
  if (!q.trim()) return 'default';
  return ID_REGEX.test(q.trim()) ? 'id' : 'name';
}

// El pill cambia con cada pulsación (sin debounce).
// El llamado a la API sí tiene debounce de 350ms.
// backend: teamtailor.service.ts
const ID_REGEX = /^(TT-?)?\d+$/i;

async searchCandidates(q?: string) {
  if (!q?.trim()) return this.getLatest(10);

  const trimmed = q.trim();
  if (ID_REGEX.test(trimmed)) {
    const numId = trimmed.replace(/^TT-?/i, '');
    return this.queryTT({ 'filter[id]': numId });
  }
  return this.queryTT({ 'filter[name]': trimmed });
}

Flujo completo de importación

  • 1

    Sin búsqueda (estado inicial)

    Carga automática de los 10 candidatos más recientes de TT al abrir el panel.

  • 2

    Usuario escribe

    El pill cambia de modo inmediatamente (sin delay). El timer de 350ms se resetea con cada pulsación. Al cumplirse, se llama a /api/teamtailor/candidates?q=texto.

  • 3

    NestJS detecta el modo

    Regex en el backend decide si usa filter[name] o filter[id]. El API Token de TT nunca sale al browser.

  • 4

    Usuario selecciona → Importar

    POST /api/teamtailor/import/:ttId. NestJS obtiene el candidato con include=documents, extrae la URL S3, descarga el CV y lo pasa al ParserService. El editor abre con los datos extraídos.

Si el candidato en TT no tiene CV adjunto: El endpoint retorna 422 Unprocessable Entity con el mensaje "Este candidato no tiene CV adjunto en Teamtailor."

Generación de PDF — Puppeteer

Diseño visual inmutable: El PDF conserva exactamente el diseño actual de Konexa (dark header, sidebar verde, tech bubbles). La plantilla Hunting.html se migra sin modificaciones al motor Puppeteer. El PDF se genera bajo demanda y se entrega directamente como descarga — no se guarda en servidor.

Flujo de generación

  • 1

    Solicitud de descarga

    Frontend llama a POST /api/candidates/:id/download-pdf. NestJS carga el candidato + plantilla de BD.

  • 2

    Compilación de la plantilla

    Handlebars inyecta los datos del candidato en el html_content de la plantilla Hunting.

  • 3

    Puppeteer renderiza

    Chromium headless carga el HTML, espera networkidle0 (para que carguen las fuentes Ubuntu), y genera el PDF A4.

  • 4

    Stream directo al browser

    NestJS retorna el buffer con Content-Disposition: attachment; filename="Juan_Perez_CV.pdf". Browser descarga directamente. Ningún archivo queda en el servidor.

pdf.service.ts

async generatePdf(id: string): Promise<Buffer> {
  const candidate = await this.repo.findOneOrFail({
    where: { id }, relations: ['template']
  });

  const html = Handlebars.compile(
    candidate.template.html_content
  )(candidate.raw_json);

  const browser = await puppeteer.launch({
    executablePath: '/usr/bin/chromium-browser',
    args: ['--no-sandbox', '--disable-setuid-sandbox']
  });
  const page = await browser.newPage();
  await page.setContent(html, { waitUntil: 'networkidle0' });
  const pdf = await page.pdf({
    format: 'A4',
    printBackground: true,
    margin: { top: '18mm', right: '18mm',
              bottom: '18mm', left: '18mm' }
  });
  await browser.close();
  return pdf; // Buffer → stream al frontend
}

El path de Chromium (/usr/bin/chromium-browser) puede parametrizarse en .env con CHROMIUM_PATH para adaptarse al servidor.

Soluciones a las 4 Vulnerabilidades Identificadas

✅ IndexError en nombre de archivo

Causa: nombres.split()[0] crash con nombres vacíos.

function buildFilename(bio: BioDto): string {
  const fn = bio.nombres?.split(' ')[0] ?? 'Candidato';
  const ln = bio.apellidos?.split(' ')[0] ?? 'Konexa';
  return `${fn}_${ln}_CV.pdf`;
}

Además el DTO marca @MinLength(1) en ambos campos antes de persistir, impidiendo guardar candidatos sin nombre.

✓ Fallback seguro + validación en DTO

✅ Dependencia de Microsoft Word

Causa: docx2pdf requiere MS Word instalado localmente.

async docxToPdf(inputPath: string): Promise<string> {
  const outDir = path.dirname(inputPath);
  await execAsync(
    `libreoffice --headless --convert-to pdf \
      --outdir "${outDir}" "${inputPath}"`
  );
  return inputPath.replace(/\.docx$/i, '.pdf');
}

LibreOffice instalado en el servidor como paquete del sistema. Ningún usuario necesita tener Office en su computador.

✓ LibreOffice headless en el servidor

✅ Colisión de hilos / bloqueo de GUI

Causa: Múltiples hilos de Tkinter competían por grab_set(), bloqueando la GUI.

Node.js es single-thread con event loop. No hay hilos de UI. Los parsings simultáneos son promesas concurrentes controladas:

async parseMultiple(files: Express.Multer.File[]) {
  return Promise.allSettled(
    files.map(f => this.parseSingle(f))
  );
  // Errores por archivo son aislados.
  // Un CV mal procesado no afecta a los demás.
}
✓ Eliminado por modelo asíncrono de Node.js

✅ API Keys en texto plano

Causa: Claves de OpenAI guardadas como JSON sin cifrar en el disco del usuario.

# .env del servidor (gitignored, chmod 600)
OPENAI_API_KEY=sk-proj-...
AZURE_CLIENT_ID=xxxxxxxx-xxxx-...
AZURE_CLIENT_SECRET=xxxxxxxxxxxxxxxxx
AZURE_TENANT_ID=xxxxxxxx-xxxx-...
TEAMTAILOR_API_TOKEN=Bearer XXXX...

CHROMIUM_PATH=/usr/bin/chromium-browser

Solo el administrador del servidor tiene acceso al .env. Ningún usuario de la aplicación ve ni gestiona credenciales.

Traefik → PM2 (host)

En la configuración dinámica de Traefik, el servicio del CV Parser apunta al host del servidor. La forma exacta depende de cómo esté configurado Traefik en el servidor:

# traefik/dynamic/cvparser.yml
http:
  routers:
    konexa-cv:
      rule: "Host(`cvparser.konexa.cl`)"
      service: konexa-cv-svc
      entryPoints: [websecure]
      tls:
        certResolver: letsencrypt

  services:
    konexa-cv-svc:
      loadBalancer:
        servers:
          # IP interna del host o host.docker.internal
          - url: "http://172.17.0.1:3001"
Nota: La IP 172.17.0.1 es la gateway por defecto de Docker hacia el host en Linux. Confirmar con ip route | grep default en el servidor.

Setup de App Registration en Azure Entra ID

Pasos para crear la App Registration dedicada a este sistema. Este proceso se realiza una sola vez y es independiente del registro de otros servicios de Konexa en Azure.

    Acceder a Azure Portal

  • 2

    Configurar nombre y tenant

    Name: Konexa CV Parser. Supported account types: elegir "Accounts in this organizational directory only" (Solo el tenant de Konexa).

  • 3

    Agregar Redirect URI

    Tipo: Web. URI: https://cvparser.konexa.cl/api/auth/microsoft/callback. Agregar también http://localhost:3001/api/auth/microsoft/callback para desarrollo local.

  • 4

    Registrar

    Clic en Register. Copiar el Application (client) ID y el Directory (tenant) ID que aparecen en el overview.

Parte 2 — Crear Client Secret

  • 1

    Ir a Certificates & Secrets

    En el menú lateral de la app recién creada → Certificates & secrets → + New client secret.

  • 2

    Configurar expiración

    Descripción: konexa-cv-parser-prod. Expiración: 24 months (máximo recomendado). Clic en Add.

  • 3

    Copiar el secret inmediatamente

    El Value solo se muestra una vez. Copiarlo ahora al .env como AZURE_CLIENT_SECRET. Si se pierde, hay que crear uno nuevo.

⚠ Recordatorio importante: El Client Secret expira. Anotar la fecha de vencimiento en un lugar visible (ej: Notion del equipo). Cuando expire, la autenticación de todos los usuarios fallará hasta que se renueve.

feat(auth): implement Microsoft OAuth2 with passport-azure-ad
AuthModule con estrategia Azure AD. Endpoints /auth/microsoft y /auth/microsoft/callback. Upsert de usuario en BD tras login exitoso.
Abrir http://localhost:3001/api/auth/microsoft → redirige a Microsoft. Ingresar cuenta corporativa → callback recibe el token → responde con JWT. Verificar en jwt.io que el payload tiene sub, email, name, role. Intentar login con cuenta externa al tenant → Microsoft rechaza el acceso.
feat(auth): add JwtAuthGuard as global guard
Guard JWT aplicado globalmente. Decorador @Public() para excluir /auth/* y /health.
curl /api/candidates sin token → 401. Con JWT válido → 200. Con JWT manipulado (cambiar un carácter) → 401. Con JWT expirado → 401.
feat(frontend): login page with Microsoft auth flow
Página /login con botón "Entrar con Microsoft". Auth context en React. Axios interceptor con el token. Ruta /dashboard protegida.
Acceder a /dashboard sin sesión → redirige a /login. Login con Microsoft → llega al dashboard. Recargar la página → mantiene la sesión (token en memoria sobrevive a renders). Cerrar sesión → vuelve a /login y el token queda borrado.

Motor de Parsing de CVs
Semanas 3–4
feat(parser): file upload endpoint (PDF + DOCX) with Multer
POST /api/parser/upload. Validación MIME y tamaño (20MB). Almacenamiento temporal en /tmp.
curl -F "file=@cv.pdf" .../parser/upload -H "Authorization: Bearer TOKEN" → 200 con tmp path. Subir PNG → 400 Unsupported file type. Subir archivo 25MB → 413 File too large.
feat(converter): DOCX-to-PDF via LibreOffice CLI
ConverterService que ejecuta LibreOffice headless. Solo se invoca para DOCX.
Subir DOCX real → logs muestran comando LibreOffice ejecutado → PDF generado en /tmp. Subir PDF directamente → LibreOffice NO se invoca (verificar en logs). Subir DOCX corrupto → error controlado, sin archivos huérfanos en /tmp.
feat(openai): CV extraction service with Files API
OpenAIService. Sube PDF a files.create, llama a responses.create con el prompt original. Parsea y valida el JSON.
Upload de un CV real completo. Verificar en logs: (1) file_id recibido de OpenAI, (2) respuesta es JSON válido, (3) JSON contiene los 5 campos raíz. Verificar que los archivos temporales se limpian de /tmp en éxito y en fallo (try/finally).
feat(candidates): persist extracted JSON to PostgreSQL
Guardar candidato con relaciones. El endpoint retorna el objeto completo. Nombre de archivo PDF validado con fallback.
Upload completo de CV. Verificar en BD: SELECT nombres, apellidos, array_length(stack,1) FROM candidates ORDER BY created_at DESC LIMIT 1;. Verificar experiencias: SELECT COUNT(*) FROM experiences WHERE candidate_id='UUID';. Subir CV con nombre vacío → 422 por validación DTO.

Frontend — Dashboard y Editor
Semanas 6–7
feat(frontend): candidates dashboard with list and search
Vista principal con tabla de candidatos. Búsqueda por nombre. Paginación. Botones de editar y descargar PDF.
Con 5 candidatos en BD, el dashboard los muestra todos. Búsqueda por nombre filtra correctamente. Clic en "Descargar PDF" → browser descarga el archivo. La tabla se refresca automáticamente tras parsear un nuevo CV sin recargar la página.
feat(frontend): CV upload panel with progress (single + bulk)
Dropzone con drag & drop. Múltiples archivos simultáneos. Barra de progreso por archivo. Lista de resultados con status.
Subir 1 PDF → progreso completa → aparece en el listado. Subir 3 archivos (2 PDF + 1 DOCX) → 3 barras de progreso individuales → 3 candidatos nuevos en BD. Subir un PNG → mensaje de error claro, el resto de archivos se procesan normalmente.
feat(frontend): interactive CV editor with form validation
Editor con formulario por secciones. Lista editable de experiencias con orden arrastrable. Stack como chips editables. Botón "Guardar" y "Descargar PDF".
Editar nombre → guardar → recargar → cambio persiste en BD. Reordenar experiencias → guardar → verificar orden en BD. Descargar PDF tras editar → PDF refleja los cambios. Dejar nombres vacío → validación en formulario impide guardar con mensaje de error inline.

Integración Teamtailor
Semana 8
feat(teamtailor): search proxy with name/ID auto-detection
TeamtailorModule con GET /api/teamtailor/candidates. Regex /^(TT-?)?\d+$/i para detectar modo. Sin params → últimos 10.
curl .../teamtailor/candidates → lista los 10 más recientes de TT. ?q=Carlos → resultados con "Carlos" en el nombre. ?q=4821 → solo ese candidato por ID. ?q=TT-4821 → mismo resultado que 4821. ?q=123abc → búsqueda por nombre (no por ID). Verificar que el TEAMTAILOR_API_TOKEN no aparece en ninguna respuesta JSON.
feat(teamtailor): CV download and import pipeline
POST /api/teamtailor/import/:ttId. Descarga CV desde URL S3. Pasa al ParserService. Guarda tt_candidate_id en BD.
Importar candidato de TT que tiene CV adjunto → candidato aparece en BD con tt_candidate_id poblado → editor se abre con datos extraídos. Importar candidato sin CV adjunto en TT → responde 422 con mensaje claro. Importar el mismo candidato dos veces → el tt_candidate_id es único, la BD acepta ambos (son instancias independientes del parsing).
feat(frontend): Teamtailor search panel with debounce + mode pill
Panel de búsqueda con input, pill de modo (nombre/ID/últimos10), debounce de 350ms, lista de resultados y botón "Importar".
Abrir panel → carga los últimos 10 automáticamente. Escribir "Mar" → pill cambia a "Búsqueda por nombre" antes de hacer el request. Esperar 350ms → resultados actualizados. Escribir "4821" → pill cambia a "Búsqueda por ID". Escribir 6 letras rápidamente → verificar en Network tab que solo se hace 1 llamada total. Clic en "Importar" → editor se abre con los datos del candidato TT.

Filtro global que intercepta todas las excepciones y retorna JSON uniforme: {"error":"...", "message":"...", "statusCode":400}. Stack traces solo en logs.

fix(parser): guarantee /tmp cleanup on all error paths
Asegurar try/finally en todos los caminos de parseSingle. Archivos huérfanos en /tmp eliminados incluso si OpenAI falla.
Simular fallo de OpenAI con API key inválida. Verificar que /tmp no contiene archivos después del error. Simular fallo de LibreOffice (archivo DOCX corrupto). Verificar mismo resultado.

Despliegue completo en el servidor. Configuración de Traefik, HTTPS con Let's Encrypt, pm2 startup.
Flujo completo en producción: (1) Login con cuenta Microsoft real → éxito. (2) Subir CV PDF real → parsear → verificar JSON en BD → descargar PDF → comparar visualmente con el PDF generado por el sistema Python actual. (3) Buscar candidato en Teamtailor por nombre → importar → descargar PDF. (4) Reiniciar el servidor → PM2 levanta el servicio automáticamente → funciona sin intervención manual.
